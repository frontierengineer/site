Install Frontier

Docker is the only dependency — the server, its worker, and the voice engines ship in one image. An NVIDIA GPU improves voice transcription; everything runs on CPU without one.

01Prerequisites

Docker, on the machine that will run the server.

curl -fsSL https://get.docker.com | sh

Or use your distribution's packages — see the Docker Engine install docs.

Install Docker Desktop for Mac and launch it once.

Install Docker Desktop for Windows and launch it once.

Docker Desktop runs containers in WSL 2; its installer sets that up.

02Pick a channel

The selected channel sets the image tag in every command below. A running install polls its channel's manifest daily and shows an in-app notice when a new build is published.

03Run a quick demo

One command, no volumes, nothing to clean up:

docker run --rm -p 61815:80 ghcr.io/frontierengineer/frontier:stable

Open http://localhost:61815. With no volume mounted the container boots in demo mode, seeded with sample data. Nothing persists once the container is removed.

04Run it for real

One image, three ways to run it — pick yours. Docker is the recommended install.

Install

docker run -d --name frontier \
  --restart unless-stopped \
  -p 61815:80 \
  -e FRONTIER_DIR=/data/frontier \
  -e FRONTIER_CHANNEL=stable \
  -v /path/to/frontier-data:/data/frontier \
  -v /path/to/frontier-home:/root \
  -v ~/repos:/workspace/repos \
  -v ~/notes:/workspace/notes \
  ghcr.io/frontierengineer/frontier:stable
  • Replace /path/to/frontier-data with the directory where you want all of Frontier's data to live. Back that directory up and you've backed up Frontier.
  • Replace /path/to/frontier-home with a directory for the container's home — keeping it means logins and caches survive recreations.
  • ~/repos and ~/notes make your local directories available to work on in Frontier. Mount as many as you like — each shows up as a place to create workspaces.

Open http://localhost:61815, then authenticate the agent before the first turn.

Adding GPU capability

Parts of Frontier — voice and transcription — work best with a GPU. Without one they degrade gracefully and everything runs; a GPU is optional. Acceleration is NVIDIA (CUDA) only today.

With an NVIDIA GPU and the NVIDIA Container Toolkit installed, add --gpus all — the same command, one new line: 

docker run -d --name frontier \
  --restart unless-stopped \
  --gpus all \
  -p 61815:80 \
  -e FRONTIER_DIR=/data/frontier \
  -e FRONTIER_CHANNEL=stable \
  -v /path/to/frontier-data:/data/frontier \
  -v /path/to/frontier-home:/root \
  -v ~/repos:/workspace/repos \
  -v ~/notes:/workspace/notes \
  ghcr.io/frontierengineer/frontier:stable

Updating

Pull the new build and recreate the container. Your data, home, and work directories are mounts — recreating the container loses nothing. 

docker pull ghcr.io/frontierengineer/frontier:stable
docker rm -f frontier
# run the same docker run command from Install

Install

The same container, declared in a docker-compose.yml:

services:
  frontier:
    image: ghcr.io/frontierengineer/frontier:stable
    container_name: frontier
    restart: unless-stopped
    ports:
      - "61815:80"
    environment:
      FRONTIER_DIR: /data/frontier
      FRONTIER_CHANNEL: stable
    volumes:
      - /path/to/frontier-data:/data/frontier
      - /path/to/frontier-home:/root
      - ~/repos:/workspace/repos
      - ~/notes:/workspace/notes
  • Replace /path/to/frontier-data with the directory where you want all of Frontier's data to live. Back that directory up and you've backed up Frontier.
  • Replace /path/to/frontier-home with a directory for the container's home — keeping it means logins and caches survive recreations.
  • ~/repos and ~/notes make your local directories available to work on in Frontier. Mount as many as you like — each shows up as a place to create workspaces.

A ready-to-edit copy is published at releases/install/docker-compose.yml. 

docker compose up -d

Updating

Pull the new build and recreate in one step — the mounts make it lossless:

docker compose pull && docker compose up -d

Install

kubectl apply -f https://frontierengineer.github.io/releases/install/kubernetes.yaml
kubectl port-forward -n frontier svc/frontier 61815:80

The published manifest is a namespace, two PVCs, a Deployment, and a Service — a starting point to adapt (storage class, ingress), not a prescription. It pins stable; for another channel, change the image tag and FRONTIER_CHANNEL together. The core of it: 

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: frontier-data
  namespace: frontier
spec:
  accessModes: ["ReadWriteOnce"]
  resources:
    requests:
      storage: 10Gi
# frontier-root: a second PVC, same shape, 5Gi
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: frontier
  namespace: frontier
spec:
  replicas: 1
  selector:
    matchLabels:
      app: frontier
  template:
    metadata:
      labels:
        app: frontier
    spec:
      containers:
        - name: frontier
          image: ghcr.io/frontierengineer/frontier:stable
          ports:
            - containerPort: 80
            - containerPort: 27520 # worker port
          env:
            - name: FRONTIER_DIR
              value: /data/frontier
            - name: FRONTIER_CHANNEL
              value: stable
          volumeMounts:
            - name: data
              mountPath: /data/frontier
            - name: root
              mountPath: /root
      volumes:
        - name: data
          persistentVolumeClaim:
            claimName: frontier-data
        - name: root
          persistentVolumeClaim:
            claimName: frontier-root
---
apiVersion: v1
kind: Service
metadata:
  name: frontier
  namespace: frontier
spec:
  selector:
    app: frontier
  ports:
    - port: 80
      targetPort: 80
      name: http
    - port: 27520
      targetPort: 27520
      name: worker
  • frontier-data holds all of Frontier's data. Back that claim up and you've backed up Frontier.
  • frontier-root is the container's home — keeping it means logins and caches survive pod recreations.
  • Mount the directories you want to work on into the pod under /workspace/<name> — as many as you like; each shows up as a place to create workspaces.

Adding GPU capability

Parts of Frontier — voice and transcription — work best with a GPU. Without one they degrade gracefully and everything runs; a GPU is optional. Acceleration is NVIDIA (CUDA) only today.

On a cluster with the NVIDIA device plugin, request a GPU on the container — the same Deployment, one new block: 

      containers:
        - name: frontier
          image: ghcr.io/frontierengineer/frontier:stable
          resources:
            limits:
              nvidia.com/gpu: 1
          ports:
            - containerPort: 80
            - containerPort: 27520 # worker port
          env:
            - name: FRONTIER_DIR
              value: /data/frontier

A GPU is exclusive to its container. On a single-GPU node, set the Deployment's update strategy to Recreate so a rolling update doesn't sit waiting for a second GPU.

Updating

Each channel publishes a manifest with the exact image of its current build, https://frontierengineer.github.io/releases/channels/stable.json. Roll the Deployment to it: 

kubectl set image -n frontier deployment/frontier \
  frontier="$(curl -s https://frontierengineer.github.io/releases/channels/stable.json | jq -r .image)"

Or wire that manifest into your own rollout machinery — Flux, Argo, a CronJob.

Workers in-cluster

In-cluster fleet workers skip pairing — on a trusted network they identify by a fixed uuid you set, no code in the manifest. Add a second Deployment from the same image with the daemon as its command: 

# in a second Deployment's pod spec
containers:
  - name: frontier-worker
    image: ghcr.io/frontierengineer/frontier:stable
    command: ["node", "/server/machine/daemon.js"]
    env:
      - name: FRONTIER_MACHINE_UUID
        value: <any-uuid> # the machine's stable identity; name it later in the UI
      - name: FRONTIER_SERVER_HOST
        value: frontier.frontier.svc.cluster.local

Give the worker a /root volume — the agent credential lives there. Share the server's volume if your storage class allows it, or mount its own and authenticate in that pod.

05Connect a machine

The server executes turns with a local worker out of the box. Each additional machine — a laptop, a desktop with a GPU, another server — runs the same daemon and joins with a short-lived code. The worker only ever dials out: no SSH, no inbound ports, no addresses to configure. Link resolves connectivity for you — it connects directly when there's a path and relays when there isn't, so the machine can sit on the same network or anywhere on the internet, end-to-end encrypted either way.

  1. Open Sessions → Machines in the left nav and click + Connect a machine. It shows a pairing code.
  2. On the other machine, run one of the two below with that code — it's all the worker needs to find your server. Copy the real command from the dialog; it carries the live code. 
    docker run -d --name frontier-worker --restart unless-stopped -v frontier-worker:/root \
  ghcr.io/frontierengineer/frontier:stable connect CODE
    curl -fsSL https://frontierengineer.com/connect.sh | sh -s -- CODE

    Identity lands under ~/.frontier-worker/.

  3. Watch it land. The machine appears in the fleet table named after its hostname the moment it pairs, with a quiet direct or via relay tag on how it reached you. The code is single-use and burns on first contact; you never type it again. A reboot or a flaky network just shows the machine going offline and coming back on its own.

06Authenticate the agent

Each machine runs an agent runtime to execute its turns — Claude Code and OpenCode are built in, and an extension can add others. You authenticate it the way that tool normally logs in; Frontier never sees or stores your credentials. Do it once per machine — the login is saved in that machine's home, so it carries across restarts and updates.

A machine you already use your agent on is already logged in — nothing to do. A fresh container worker is isolated, so log in inside it once: open a shell (docker exec -it frontier-worker bash) and run your runtime's normal login there.
